# Internationalization (i18n / translation)

Status: **all eight implementation phases are built.** English plus AI-generated Preview
catalogs for French (`fr`), German (`de`), Italian (`it`), and Spanish (`es`) ship with the
browser client. Static extraction and catalog validation are complete; the remaining release
handoff is the project owner's required manual browser QA under `file://`.

This is the authoritative design for localization. It records the translatable surface, the
resolved architecture decisions, and the current AI-only pipeline. Consult it before
touching `FB.fmt` (`js/events.js`), event/data display fields, `FB.news`, or the `FB.fx`
presentation-intent outbox.

## 1. The surface

The canonical inventory is generated by `python tools/i18n_catalog.py extract`; exact
current counts live in the generated coverage report. It includes more than five thousand
owner keys, with identical English records deduplicated without coupling their future
translations. Events remain the largest prose bucket.

| Bucket | ~Strings | ~Words | Notes |
|---|---|---|---|
| Event packs (`data/events_*.js`) | ~1,092 fields | ~9,800 | `title/text/label/desc/log/worldNews`; id-keyed |
| UI — `ui.js` | ~300 | ~2,700 | welded into single-quote HTML concatenation |
| UI — `actions.js` (deeds) | ~150 | ~1,100 | `label`/`desc` per deed |
| UI — `main.js` (scenarios, help, epitaphs) | ~150 | ~1,450 | excl. changelog |
| UI — other engine JS | ~200 | ~1,300 | `world.js`, `armies.js`, `model.js`, `portrait.js` |
| `index.html` static | ~50 | ~300 | buttons, tabs, headings |
| `traits.js` (traits + ailments) | ~96 | ~380 | `name`/`desc` |
| `map_data.js` gameplay (buildings/holdings/items/plots/tech) + 56 titles | ~154 | ~1,900 | `name`/`desc`, accept tokens + variant objects |

**Optional / never:** county place-names (`counties.js`, ~498) and realm/de-jure names
(`map_data.js`, ~363) are proper nouns — localize only where exonyms genuinely differ, later.
The personal-name and settlement-fragment lists (`cultures.js`, ~1,500+) are **generation
fuel and must never be translated** — they seed procedural names baked into saves.

The changelog (`js/main.js`) is meta and stays English.

## 2. The two structural problems the implementation resolves

Machine translation is not the difficult part. The original difficulty was structural:

1. **No layer.** Much of the UI was welded into HTML built by concatenation. Builder
   chokepoints, explicit lookup calls, and an exact-source completed-DOM pass now route it.
2. **Grammar was spliced in JavaScript, not in the text.** Gender/relation ternaries, plural
   suffixes, and the former `{warstate}` sentence could not be translated safely. Complete
   selector records and message parameters now own that grammar.

What the final design preserves:

- **The `{token}` system is translation-safe.** `FB.fmt` (`js/events.js:335-426`) does a
  positional `str.replace(/\{(\w+)\}/g, …)` over ~30 tokens. `'you are wed to {suitor}'` →
  `'vous épousez {suitor}'` keeps the token; `FB.fmt` fills it identically. Translators keep
  the braces; word order around them is theirs to set.
- **The `{default, muslim, jewish}` variant-object convention** remains authoritative source
  data. The renderer selects the source branch before looking up its localized shadow record.
- **A few centralized maps exist as anchors:** `FB.SKILL_NAMES` (`model.js:9`), `FB.SEASONS`
  (`util.js:57`), `MAPMODES` (`ui.js:759`), `FBDATA.titles` (`map_data.js:606-614`).

## 3. Architecture

### Catalog shape and authority

**Locale files are `.js` globals, not JSON.** This is forced by the `file://` constraint:
`fetch()` of a local file is blocked in Chrome, which is why the whole data layer uses script
tags. Each locale is one self-describing, schema-versioned catalog:

```js
(function () {
  var R = { /* generated deduplicated {text,hash} / {forms,hash} records */ };
  FBDATA.lang.fr = {
    schema: 1,
    code: 'fr',
    name: 'Français',
    dir: 'ltr',
    entries: { /* namespaced owner key -> R[generated record id] */ },
    words: { /* isolated closed-vocabulary words only */ },
    pluralCategory: function (n) { return n >= 0 && n < 2 ? 'one' : 'other'; }
  };
})();
```

`entries` is the runtime flat namespaced owner-key map. The generated file deduplicates
identical target records internally, but every key keeps an independent mapping, so two
contexts with the same English today may diverge in a later translation. A record is
`{text, hash}` or `{forms, hash}`; raw catalog strings are invalid. `forms` selects complete
phrases by an explicit numeric plural parameter or exact semantic value, and may nest one
further selector only when two dimensions genuinely combine. A target locale may add
`zero`/`two`/`few`/`many` plural cases that English does not need; each added case is
validated against English’s `other` leaf for nested shape, placeholders, and emoji.
Exact-value selectors may not invent semantic cases.

English remains authoritative in the source data and readable UI calls. A generated
`data/lang_en.js` source manifest mirrors the effective English surface for extraction,
context, validation, and translation work; it is not a second hand-maintained source of
truth. Opaque durable-message keys additionally have a shared English registry so
`{key, params}` always has a readable fallback.

### Lookup strategies

All localization uses a **shadow lookup**. Never mutate `FBDATA` display fields: the same
English source remains available to the simulation, source hashing, mods, and fallback
logic.

- **Structured data (events, traits, buildings, items) → owner-aware id-keyed accessors.**
  These already have stable ids (`id:'meet_suitor'`). Keys include the owner namespace, id,
  authored field path, and selected source variant. Event option paths always use the
  original authored option index, never the filtered visible position. The English source
  remains untouched in the data files.
- **UI strings → English-as-key.** Wrap ordinary chrome with
  `FB.T('Reputation among the folk')`. Use `FB.TC(context, 'English text')` when the same
  English wording has different meanings or grammatical roles. A miss falls back to the
  English argument automatically.
- **Durable/shared messages → opaque namespaced keys.** Chronicle entries, `FB.fx`
  presentation intents, and other durable messages use a registry key such as
  `news.marriage.child`, never an English sentence as the durable identifier.

### Resolution order

Structured text resolves in this order:

1. Identify the explicit viewing player/owner; do not infer the viewer from the singleton
   `state.player`.
2. Read the authoritative English source field and select its faith/value branch
   (`muslim`, `jewish`, or `default`).
3. Derive the exact namespaced key for that selected branch and look it up in the locale
   shadow catalog.
4. Reject a missing, invalid, or source-hash-stale record and fall back to that exact English
   branch.
5. Apply structured plural/value selection.
6. Substitute plain `{identifier}` placeholders from explicit display parameters.

This branch-first rule prevents a translation from accidentally erasing faith flavor and
lets each branch carry its own source hash.

### Locale lifecycle and browser safety

- A fixed client-side locale manifest lists every selectable supported or preview locale,
  including code, file, native name, direction, and status. The chooser reads this manifest;
  it does not enumerate whichever `FBDATA.lang` objects happen to be loaded. Locale codes
  never flow unchecked into a script path.
- At boot, read the display-only locale preference from `localStorage`, validate it against
  the manifest, load that script before the first localized render, apply mods, then validate
  the catalog and effective-source hashes. Any load or validation failure falls back cleanly
  to English and records English as the active locale.
- Changing language persists the selection and **reloads the page**. A reload is the safe
  switch for title screens, cached panels, canvas labels, and an active save; do not maintain
  a second live-switch invalidation system. The chooser is available on the title screen and
  in the in-game Settings dialog; an active life autosaves before reload, while observe mode
  remains intentionally unsaved.
- Keep committed `file://` script URLs query-free. Under HTTP(S), append
  `?v=<FB.VERSION>` to dynamically loaded locale scripts so a release cannot reuse a stale
  catalog. This mirrors deploy cache-busting without breaking local-file play.
- Set `<html lang>` and `dir` from the validated catalog. Direction is catalog metadata, not
  inferred from the locale code.

### UI insertion and static HTML

Catalog strings are untrusted **plain text**, not HTML. DOM code uses `textContent`; HTML
builders escape the localized result exactly once at the insertion boundary. Semantic
message params stay raw until that boundary — callers do not pre-escape values stored in
state. Markup-bearing English fragments must be split into DOM structure plus localizable
text rather than placed in a catalog record.

Static `index.html` uses explicit `data-i18n`, `data-i18n-title`, and
`data-i18n-aria-label` annotations while preserving English in the committed HTML as the
no-script/fallback source. Completed dynamic trees may also receive `FB.localizeTree`: it
only replaces an entire text node or attribute whose exact English source is known. It never
does substring, fragment, name, or numeric replacement, and `data-i18n-ignore` opts a subtree
out.

**Invariant that keeps this safe: only pure-display fields are localized** — `title`, `text`,
`label`, `desc`, `log`, `name`. Ids, effects, triggers, and numbers are **never** touched.
This is what lets a localized view coexist with the deterministic sim: text is display-only,
so swapping the language can never change simulation outcomes.

**Do not build ICU MessageFormat.** It is unnecessary for this game and conflicts with the
zero-dependency target. Structured complete-phrase selectors cover the required plural and
value cases without putting grammar inside an inline mini-language.

The first selected set is French, German, Italian, and Spanish. All four are LTR Latin-script
catalogs and are labelled **Preview** because their translation is AI-only for now. CJK layout
and RTL/bidi hardening are separate later scopes; selecting a language that needs either must
explicitly schedule that work rather than quietly declaring it unsupported.

## 4. Mechanics: one message layer

Both structural problems (§2) collapse into **one mechanism**, applied at **builder
chokepoints** rather than per literal. A message is `{key, params}`; rendering is lookup,
structured selection, and plain substitution.

### Problem 1 (no layer) → localize at chokepoints, because the concatenation is regular

The ~300 `ui.js` strings flow through ~5 structural patterns (`ui.js:388-401` is almost all
`<span>LABEL</span><b>VALUE</b>` rows). Localize the patterns, not the strings:

```js
// before (ui.js:393)
'<div class="kv"><span>Reputation among the folk</span><b>' + Math.round(s.player.pop) + '</b></div>'
// after — kv() wraps FB.T once and covers every label row
kv('Reputation among the folk', Math.round(s.player.pop))
```

`toast()` is already one function — route it through `FB.T` and every toast is covered at a
single site; likewise `panelh('Skills')` for headers, a `data-i18n` pass for the ~50 static
`index.html` strings, and direct `FB.T('…')` for the residue. When a string has spliced
values, **wrapping and templating are one edit**:

```js
'You have ' + n + ' children'   ->   FB.T('You have {n} children', { n: n })
```

(Reject substring DOM post-processing — it translates values/names blindly and cannot reorder
words. The implemented exact-source completed-node pass is deliberately narrower: it only
handles a whole known English text node or translatable attribute. Templates and builder
chokepoints remain mandatory for mixed text.)

### Problem 2 (spliced grammar) → complete-phrase selectors, no ICU

- **Sentence grammar → structured plural/value records.** Replace suffixes and ternaries
  such as `'compan'+(n>1?'ies':'y')` and
  `(sex==='f'?'daughter':'son')` with catalog records whose forms are complete phrases.
  English needs plural categories `one` and `other`; languages that require them can supply
  `few` and `many`. Exact semantic selection uses stable values such as `m`, `f`, or a
  relationship id. Translators own the complete word order in every form.
- **Isolated vocabulary → `FB.word(concept, features)`.** Keep a small per-locale lexicon for
  standalone titles, relationship labels, and other words that genuinely appear outside a
  sentence. Do not use `FB.word` to splice grammar into prose. Precedent exists in
  `FBDATA.titles` and `FB.SKILL_NAMES`.
- **Assembled sentences → `{key, params}` + a catalog record.** This covers the structured
  war-status clauses, every core `FB.news` producer, and legend title/quip assembly. Text
  supports plain `{identifier}` placeholders only; there is no typed placeholder
  mini-language. Parameters contain JSON-safe semantic values, ids, generated proper names,
  and numbers — not rendered localized fragments.

**`FB.news` was the highest-value conversion.** Core callers now store locale-neutral
descriptors in `state.log`; the string form remains only for compatibility with old saves and
unstructured third-party mod code:

```js
// legacy, retained for old saves and gradual conversion
FB.news(state, 'The harvest fails.');
// state.log entry: { y, s, d, t: 'The harvest fails.' }

// durable form — the object makes it unambiguous that this is not rendered prose
FB.news(state, { key: 'news.harvest.failed', params: {} });
// state.log entry: { y, s, d, msg: { key: 'news.harvest.failed', params: {} } }
```

Two payoffs from that one signature: the chronicle **re-renders in the player's current
language** instead of freezing; and save-compat is free — the display path renders `msg` if
present, else the legacy `t` string, so old saves stay English with no migration (recorded in
[state-and-saves.md](state-and-saves.md)). Item, building, plot, recipient, gold, title, and
province values are carried as semantic parameters rather than pre-rendered localized
fragments.

**Legends split across both mechanisms:** trait/skill quip tables are id-keyed source data,
while conditional wrapper sentences are opaque messages. New legend entries store
`titleData`, `causeMsg`, and `quipMsg` so titles, causes, and quips render in the currently
selected locale. Legacy rendered legend fields still display unchanged.

### The unification — one core, three entry points, structured selectors

```
message       = { key, params }
record        = lookup(locale, key) || englishTemplate
template      = select(record, locale, params)  // optional plural/value forms
render(msg)   = substitute(template, params)
substitute:     {name} {suitor} {holy} <- plain values from explicit display params
```

Plural and gender/value choices live outside the prose as structured message records, not as
an inline mini-language. A record may select on a numeric parameter using the locale's
`one/few/many/other` plural rule, or on an exact semantic value such as `m`/`f`; a form may
contain another selector only when two dimensions genuinely combine. Translators own each
complete phrase. Text itself supports plain `{identifier}` placeholders only. `FB.word`
remains for isolated vocabulary such as standalone titles or relationship labels; sentence
grammar uses complete selector forms.

`FB.T`/`FB.TC` (UI), owner-aware structured-data accessors, and durable-message rendering
become entry points onto the **same lookup + selector + plain-substitution core**. Target
signatures for the plumbing:

```
FB.T(str, params?)             -> lookup('ui:'+str) || str, then select + substitute
FB.TC(context, str, params?)    -> contextual UI key, then the same rendering core
FB.eventText(state, viewer, ev, path, ctx)
                               -> select source branch, derive event.<id>.<path>.<branch>,
                                  translate that exact branch, then substitute explicit ctx
FB.dataText(state, viewer, kind, id, def, path, ctx?)
                               -> derive owner-aware namespaced key, then select + substitute
FB.news(state, message|string) -> store {key,params} message, or legacy rendered string
FB.word(concept, features)     -> isolated vocabulary only; features are JSON-safe semantics
FB.pluralCategory(locale, n)   -> one/few/many/other for structured plural selectors
FB.renderMessage(message, ctx) -> render an opaque registry/catalog message without mutation
```

At every durable-message boundary, clone and validate `params` as JSON-safe data and freeze
the in-memory descriptor in development builds. Never retain a mutable caller-owned object
in saved state.

### Implemented layering

1. **Durable core:** `js/messages.js` owns descriptor validation/cloning, the opaque English
   registry, `FB.news`, and descriptor-shaped `FB.fx` intents without depending on a locale.
2. **Browser renderer:** `js/i18n.js` owns catalogs, selectors, substitution, shadow
   data/event accessors, the fixed manifest, diagnostics, and browser loading.
3. **Chokepoints and exact routing:** toast, modal, row/header builders, static attributes,
   and exact-source completed-tree localization cover regular UI; explicit templates cover
   mixed text.
4. **Grammar and saved prose:** structured selector/message records replace suffixes and
   ternaries; chronicle and legend entries keep semantic descriptors.

All new durable/shared message sites must use descriptors. Local UI chrome may continue to use
English-as-key, but text stored in state or emitted as a presentation intent may not be baked
in the active locale.

## 5. Staleness & maintenance

We ship versions constantly (1.17.2 already). The instant an English string changes, its
translation rots silently. Two mechanisms, and one falls out for free:

- **English-as-key self-heals.** Change the English → the key changes → automatic miss →
  falls back to the *new* English. No stale translation is ever shown. (Downside: the old
  translation is orphaned until re-linked — acceptable.)
- **Id-keyed data and opaque messages need an explicit source hash.** Their keys stay stable
  while English can change underneath. Hash a canonical representation containing the hash
  schema/version, selected English source record, selector structure, and placeholder set.
  Compute expected hashes from the **effective English data after runtime mods are applied**.
  On mismatch, reject that localized record and fall back to current English.

Maintenance loop = each release, extract English, AI-translate only new or source-changed
records, then run structural/hash/token/emoji/completeness validation. A
**pseudo-localization dev mode** (accented look-alike chars, roughly +30% length) surfaces
missed routing and layout breaks; German-length expansion catches tight controls early.
Development diagnostics and a copy-key aid identify a failing owner key precisely.

**Supported-locale release gate:** a locale appears as supported only when **100% of the
in-scope core catalog** is translated, source-hash-current, structurally valid, token-valid,
and routed through the localization layer. Incomplete locales remain development-only or are
explicitly labelled **Preview**. Runtime English fallback remains a safety net, but English in
ordinary core UI or events under a supported locale is a bug. Explicit exceptions are
generated/player-authored names, currently deferred county/realm proper nouns, untranslated
mods, legacy rendered strings in old saves, and the English-only changelog.

The four current machine-generated catalogs can pass every automated structural gate and
still remain **Preview**: automated validation proves integrity and completeness, not
linguistic quality. Promotion to Supported is deferred until the owner later chooses an
appropriate quality-review process.

## 6. The AI-only pipeline (current policy)

1. **Extract** the canonical English catalog and tool manifest with every owner key, source
   record, context, location, token/emoji set, and source hash.
2. **AI-translate `fr`, `de`, `it`, and `es`** in coherent cached batches. Preserve every
   `{token}`, emoji, selector shape, and the literary register; reuse the cache when English
   has not changed.
3. **Validate automatically:** exact key coverage, catalog/schema/code metadata, record
   identity, source hashes, selector structure, required fallbacks, tokens, emoji, and
   orphaned records. A failed locale is not activated and English remains safe.
4. **QA structure and layout:** use the pseudo-locale and copy-key diagnostics, then perform
   the repository-required manual browser pass. No human linguistic review is required in
   the current workflow.

This deliberately trades review time for speed. The four catalogs remain labelled
**AI Preview** even at 100% automated coverage. Human/native-speaker review can be added
later without changing catalog keys or architecture; only then should a Preview locale be
considered for Supported status.

## 8. Implementation status and closed decisions

The production-shaped implementation is present:

- `js/messages.js`: locale-neutral descriptors, English registry, JSON-safe cloning/freezing,
  `FB.news`, and `FB.fx`.
- `js/i18n.js`: manifest/loading/fallback, schema and effective-source validation, source
  hashes, selectors, substitution, exact-source DOM routing, diagnostics, copy-key support,
  owner-aware event/data accessors, and pure descriptor rendering.
- `data/lang_en.js`: generated canonical English catalog; `data/lang_qps.js`: development
  pseudo-locale; `data/lang_{fr,de,it,es}.js`: AI Preview catalogs.
- `tools/i18n_catalog.py`: static extraction only (it never executes the game), deterministic
  manifest generation, translation cache integration, and structural/completeness validation.
- Core UI, events, actions, structured data, chronicle, toasts, death/legend records, save
  labels, and title snapshots route through the layer. Proper names and documented exceptions
  remain English by design.

All previously open design choices are closed:

- Shadow lookup; never mutate source `FBDATA` display fields.
- Load only the selected locale dynamically from a fixed manifest.
- Persist locale outside saved/seeded state and switch it by page reload.
- Ship French, German, Italian, and Spanish together as AI-only Preview locales.
- Do not require human translation or linguistic review in the current workflow; keep the
  Preview label until a later quality-review decision.
- Defer county/realm names; later add only useful exonyms. Never translate name-generation
  fuel.
- English remains authoritative; generate `lang_en.js` for tooling and keep an opaque-key
  English registry for durable messages.
- Select the source faith branch first, then translate and hash that exact branch.
- Use complete structured plural/value forms; reserve `FB.word` for isolated vocabulary.
- Require 100% current in-scope coverage before marking a locale supported; otherwise label
  it Preview/development-only.
- Keep the simulation locale-neutral and rendering pure; localize independently in the
  browser client.
- Explicit English exceptions: generated/player-authored and deferred geographic proper names,
  unrecognized mod-authored text, legacy rendered prose already stored in old saves, and the
  changelog.

## 10. Detailed eight-phase implementation record

This checklist records what was implemented and the acceptance criteria that future changes
must preserve.

### Phase 1 — Freeze the durable contracts

Status: **complete.**

Goal: make the data that crosses saves stable before expanding UI coverage.

1. Split the logical responsibilities:
   - A versioned message contract defines descriptor validation, opaque-key English records,
     `FB.news`/`FB.fx` semantics, and simulation-time context materialization.
   - Browser code implements that contract and owns locale loading, catalogs, source hashes,
     lookup, selection, substitution, DOM localization, and final rendering.
2. Make `{key, params}` the one durable message shape. `FB.news` accepts either the object or
   a legacy rendered string; do not retain the ambiguous `(key, params)` overload.
3. Keep `state.log` entries as `{y,s,d,msg:{key,params}}` or legacy `{y,s,d,t}`. Route
   structured entries through `FB.renderMessage`, not through the `ui:` namespace in `FB.T`.
4. Define namespaced opaque keys for every shared channel (`news.*`, `fx.*`, and any later
   durable notification) and register their authoritative English records.
5. Clone and validate message params at the boundary. Permit only JSON-safe semantic values,
   ids, generated proper names, booleans, numbers, strings, arrays, and plain objects. Reject
   functions, DOM/state objects, cycles, and pre-rendered localized fragments. Freeze the
   normalized descriptor in development builds.
6. Make viewer/owner identity explicit in event/data rendering APIs. Do not assume the
   singleton `state.player`; take the viewer explicitly.
7. Make rendering pure: no RNG consumption, role creation, state repair, or other mutation.
   Move the current `FB.getRole(state, role, true)` behavior out of `FB.fmt` display-time
   token resolution and into simulation-time event/context materialization.

Done when:

- The simulation can create and save descriptors without loading a locale or depending on
  `document`, `localStorage`, `FB.ui`, or `FB.map`.
- Rendering the same descriptor twice produces the same text and does not change state or RNG.
- Legacy `t` chronicle entries still render unchanged.

### Phase 2 — Replace the provisional catalog and rendering core

Status: **complete.**

Goal: implement the final schema and one rendering path before any locale depends on the
flat Phase 1 prototype.

1. Replace direct properties on `FBDATA.lang[locale]` with the schema-versioned catalog:
   `{schema,code,name,dir,entries,words,pluralCategory}`.
2. Accept only `{text,hash?}` and `{forms,hash}` entry records. Reject raw catalog strings.
3. Implement one shadow-lookup function; never mutate English fields in `FBDATA`.
4. Implement structured selector records:
   - Numeric selectors use the active catalog's `one/few/many/other` category.
   - Exact-value selectors use stable semantic values.
   - Forms contain complete phrases and may nest only one additional selector when required.
5. Remove prose construction through `FB.plural(count, forms)`. Retain
   `FB.pluralCategory(locale, n)` for the selector engine.
6. Keep `FB.word` only for isolated vocabulary, with explicit JSON-safe features and English
   fallback per concept.
7. Implement `FB.TC(context, english, params)` alongside `FB.T`; define a stable contextual
   key encoding that cannot collide with ordinary `ui:` keys.
8. Make `FB.T`, `FB.TC`, `FB.eventText`, `FB.dataText`, and `FB.renderMessage` use the same
   select-and-substitute core.
9. Treat catalog output as plain text. Substitute plain `{identifier}` placeholders only;
   preserve unknown placeholders for diagnostics and never interpret catalog HTML.

Done when:

- The old flat lookup and standalone `FB.plural` prose helper have no callers.
- A small in-memory development catalog can exercise text, plural, exact-value, nested,
  contextual, missing-key, and English-fallback paths through the same renderer.
- No real locale file has been required to validate the core.

### Phase 3 — Implement the fixed locale manifest and boot lifecycle

Status: **complete.**

Goal: make selecting, loading, persisting, and failing a locale deterministic under both
`file://` and hosted play.

1. Add a fixed manifest containing English plus explicitly registered supported/preview
   locales. Include code, script path, native name, direction, and release status.
2. Populate the Settings selector from the manifest, not from already-loaded
   `FBDATA.lang` tables. Remove arbitrary locale-code-to-path construction.
3. During boot:
   - Read `fb_lang` from `localStorage`.
   - Validate it against the manifest.
   - Load only that catalog before the first localized render.
   - Apply runtime mods, then validate catalog schema and effective-source hashes.
   - Fall back to English on any load or validation failure.
4. Change language by persisting the new code and reloading the page. Remove the provisional
   live redraw path and its incomplete cache invalidation.
5. Keep committed and `file://` script URLs query-free. Append `?v=<FB.VERSION>` only under
   HTTP(S).
6. Set `<html lang>` and `dir` from validated metadata and restore English metadata on
   fallback.
7. Surface a concise development diagnostic for load/schema failure without exposing an
   incomplete locale as supported.

Done when:

- A development-only dummy locale can be selected from the manifest, survives reload, and
  loads from both `file://` and HTTP(S).
- Missing, malformed, wrong-code, and wrong-schema catalogs reliably return to English.
